SM-IDE Help

What is SM-IDE?

SM-IDE is a browser-based player and tester for chatbot flows defined as XState v5 state machines. You supply two files — a machine definition (JSON) and a services file (JS) — and the IDE renders the chat UI, draws the state diagram, and can automatically run every possible conversation path as a regression test.

No build step, no install. Everything runs in the browser.

Loading a Flow

  1. Click Load Flow in the toolbar.
  2. In the file picker, select your machine.json. If the flow uses guards or async services, also select the services.js at the same time (hold Ctrl / Cmd to multi-select). You can also select a ZIP that contains both.
  3. The chat UI and diagram appear instantly. The machine starts from its initial state.

You can also load a ZIP archive that contains both files — use Save Flow to download one from any loaded flow.

The Machine File (*-machine.json)

A plain JSON object in XState v5 config format. The minimal keys are id, initial, context, and states.

State types at a glance

PatternMeaning
"on": { "EVENT": { "target": "…" } } Waits for a user event (button tap or text submit)
"on": { "SUBMIT": [ { "guard": "…" }, { … } ] } Validated text input — guard passes on valid value, fallback on invalid
"always": [ { "guard": "…", "target": "…" } ] Immediate transition — evaluated as soon as the state is entered
"invoke": { "src": "…", "onDone": … } Calls an async service; transitions when it resolves
"type": "final" Terminal state — conversation complete

State meta — what the chat UI reads

FieldEffect
meta.text Bot message shown when entering the state. Use {{contextKey}} to interpolate context values.
meta.input: "text" Shows a free-text input box instead of buttons
meta.placeholder Placeholder for the text input box
meta.choices Array of button labels (overrides auto-generated event names)

Example: text input with validation

// ask_email state from realtor-machine.json
{
  "ask_email": {
    "meta": {
      "text": "Welcome! Please enter your email address.",
      "input": "text",
      "placeholder": "email@example.com"
    },
    "on": {
      "SUBMIT": [
        {
          "guard": "isValidEmail",          // pass → advance
          "target": "ask_status",
          "actions": [
            { "type": "updateContext", "params": { "mapValueTo": "user_email" } }
          ]
        },
        {
          "actions": [                          // fail → show error, stay
            { "type": "updateContext", "params": {
              "inputError": "Please provide a valid email address."
            }}
          ]
        }
      ]
    }
  }
}

Example: choice buttons

{
  "ask_status": {
    "meta": { "text": "Do you rent or own your current home?" },
    "on": {
      "RENT": { "target": "ask_timing", "actions": [
        { "type": "updateContext", "params": { "status": "rent" } }, "record"
      ]},
      "OWN":  { "target": "ask_order", "actions": [
        { "type": "updateContext", "params": { "status": "own"  } }, "record"
      ]}
    }
  }
}

Example: async service invoke

{
  "send_transcript": {
    "meta": { "text": "Got it! A transcript will be emailed to {{user_email}}." },
    "invoke": {
      "src": "emailTranscript",   // name must match a key in services
      "onDone":  { "target": "upload_context" },
      "onError": { "target": "upload_context" }
    }
  }
}

The Services File (<id>-services.js)

A single ES module that exports one object containing all async service functions, guards, and sample inputs used by the machine. Name it to match your machine: if your machine id is my_bot, call it my_bot-services.js. Because the file is loaded as a blob URL at runtime, it must be fully self-contained — no relative imports.

Export your services object as a named export (export const myServices = {…}) or as the default export (export default {…}). The name does not matter — the IDE takes the first named export, or the default export if there is no named one.

Minimal structure

// my_bot-services.js — fully self-contained, no imports
export const myServices = {

  // ── Sample inputs (required if machine has text-input states) ──
  // Keyed by state id (states with meta.input === "text").
  // Used by the test runner to fill SUBMIT events during path generation.
  // Values must pass any guard on the SUBMIT transition.
  SAMPLE_INPUTS: {
    ask_email: 'test@example.com',
    ask_phone: '4155550123',
  },

  // ── Async services ─────────────────────────────────────────────
  // Each key matches an "invoke.src" name in the machine JSON.
  // Receives { input } where input is the current context.
  emailTranscript: async ({ input }) => {
    // TODO: replace with real API call
    await new Promise(r => setTimeout(r, 0)); // yield one tick
    return { ok: true };
  },

  // ── Guards ─────────────────────────────────────────────────────
  // Pure booleans. Each key matches a "guard" name in the machine.
  guards: {
    isValidEmail: ({ event })   => /^[^\s@]+@[^\s@]+\.[^\s@]{2,}$/.test(event.value),
    isSoon:       ({ context }) => context.timing === 'soon',
  },
};
Keep all helper code (validators, constants) inside the same file. Relative imports won't work because the file is loaded as a blob URL.
If the flow has no services or guards, the services file can be omitted entirely — just load the machine.json on its own.

Walkthrough: Realtor Bot Example

The flows/ folder contains a complete example — a real-estate lead-capture chatbot. Its conversation path looks like this:

ask_email
→
ask_status
→
ask_timing / ask_order
→
ask_phone
→
route_by_timing
→
closing_soon / closing_later
→
send_transcript
→
upload_context
→
finish_screen

Purple boxes are invoke states (async service calls). Green is the final state.

Guards isValidEmail and isValidPhone keep the user on the current state until valid input is provided. route_by_timing is an always state — it transitions immediately based on context.timing without user interaction.

Built-in Runtime Actions

These actions are provided automatically by the IDE runtime — you do not need to implement them in your services file. Reference them by name in your machine JSON.

ActionWhere to useWhat it does
initTrace Top-level "entry" on the machine Initialises the _trace envelope on context (flowId, sessionId, startedAt, empty steps array). Required for session recording and test results.
record "actions" on transitions Appends the current state + event to _trace.steps. Add to every transition you want captured in the transcript.
updateContext "actions" on transitions (parameterised) Merges the params object into context. Use mapValueTo as a param key to store the submitted event value into a named context field.

Example: enabling tracing and recording

{
  "id": "my_bot",
  "initial": "first_state",
  "entry": ["initTrace"],      // ← initialise _trace when machine starts
  "context": { "user_email": "" },
  "states": {
    "first_state": {
      "on": {
        "SUBMIT": [
          {
            "guard": "isValidEmail",
            "target": "next_state",
            "actions": [
              { "type": "updateContext", "params": { "mapValueTo": "user_email" } },
              "record"         // ← append step to _trace
            ]
          }
        ]
      }
    }
  }
}

Running Tests

Click Test to enumerate every possible path through the loaded machine and replay each one headlessly. Results appear in a drawer at the bottom of the screen.

  • Each row shows the expected event sequence, final state, and pass/fail.
  • Click a row to restore the chat bubbles and diagram for that path.
  • Click 💾 Download JSON in the results drawer to save the results for later.
  • Click ⛔ Stop in the status badge to abort a long run.
Tests run headlessly — one fresh runtime per path, no DOM rendering during replay. For a 24-path machine like the realtor bot this typically completes in under a second.

How paths are generated

The IDE walks the machine graph depth-first and emits one path per unique route to a final state (or per unique prefix if there is no final state). For text-input states (meta.input: "text") it needs a sample value to use as the submitted input for each such state.

Guarded invoke.onDone branches are not enumerated — only the unguarded fallback is tested, because the outcome of a real service call cannot be known statically.

Sample inputs for text states

Add a SAMPLE_INPUTS property inside your services object, keyed by state id. The test runner will block until all text-input states have a sample value and will show a dialog with the required snippet if any are missing. 

export const myServices = {

  // Keyed by state id (states with meta.input === "text").
  // The test runner submits these values on each SUBMIT step.
  SAMPLE_INPUTS: {
    ask_email:           'test@example.com',
    ask_phone:           '4155550123',
    extend_quota_check:  'extend',
    upgrade_plan_check:  'Pro',
    downgrade_plan_check:'Starter',
  },

  // … async services and guards …
};
Each value must pass any guard on the SUBMIT transition for that state — otherwise the test runner will skip that path as a validation failure.

Loading Saved Results

  1. Click Load Results in the toolbar.
  2. Select a test-results-*.json file previously downloaded from the results drawer.
  3. The IDE restores the machine, services, and all test cases. Click Test to re-run against the same machine.
Results files are self-contained — they include the machine config and the full services source so you can share them and re-run without the original files.

TypeScript Export (Stately VS Code Extension)

To edit a flow visually using the Stately VS Code extension, export the loaded flow as a ready-to-open TypeScript project: 

window.exportToTypeScript()

This downloads {id}-xstate.zip containing:

  • {id}-machine.ts — createMachine() config with inferred context and event types
  • {id}-services.ts — typed implementation stubs (or your existing services wrapped)
  • package.json — xstate ^5 + typescript ^5 dependencies
  • tsconfig.json — minimal TypeScript config for XState v5
  1. Unzip and run npm install in the folder.
  2. Open the folder in VS Code — the Stately extension will detect and visualise {id}-machine.ts automatically.
  3. Fill in the // TODO stubs in {id}-services.ts.
To strip meta fields from the export (some Stately extension versions warn about them), use: window.exportToTypeScript({ nometa: true })

Context Viewer

The collapsible panel at the bottom of the right pane shows the current XState context and a step-by-step trace of the conversation:

  • ✅ steps — valid user inputs that advanced the machine
  • ❌ steps — validation failures (user stayed on the same state)
  • ⚙️ steps — async service calls and their results

IDE Developer

This section is for developers working on the IDE itself, not on flows. All commands are run in the browser DevTools console.

Regression Contracts — quick start

Contracts are snapshot tests that verify the IDE's own UI, runtime behaviour, toolbar DOM, and load routing. Run them before and after any change to catch regressions.

For stable UI measurements, open the IDE in a fixed-size window first:
window.open(location.href, '_blank', 'width=1024,height=768')
then undock DevTools before running UI contracts.
CommandWhat it checks
await window.contracts.ui() Layout snapshots at 1024×768 — toolbar, chat pane, diagram pane
await window.contracts.ui('capture') Force-capture a new UI baseline
await window.contracts.runtime() Runtime behaviour — state transitions, context updates, guard evaluation
await window.contracts.runtime('capture') Force-capture a new runtime baseline
await window.contracts.smide() SM-IDE machine contracts — toolbar state per machine state
await window.contracts.smide('capture') Force-capture a new smide baseline
await window.contracts.load() Toolbar DOM + load routing + ZIP round-trip

Workflow:

  1. Run ('capture') once on a known-good state and commit the resulting *-baseline.json files.
  2. Run the plain () form before and after every change — any deviation from the baseline is reported as a failure.
